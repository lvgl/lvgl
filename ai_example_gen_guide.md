# LVGL XML Example Generation Guide

This guide documents all rules and best practices learned through systematic
creation of XML examples — covering feature, styling, and data-binding
examples across the converted widgets.

---

## Core Principles

### Feature-Based Coverage
- Create one XML example file per feature, not broader topics
- Examples should isolate and demonstrate a single configurable aspect
- Group examples only when direct visual comparison is essential (e.g., arc modes, slider modes)

### Avoid Repetition
- Do not show default values or "baseline then varied" comparisons unless the default illustrates a limitation
- Remove defaults from attributes (e.g., `bg_start_angle`, `bg_end_angle` for arcs when they match defaults)
- Skip redundant demo items; focus on demonstrating the feature difference only

---

## Documentation Structure

### Example File Organization
- **Path**: `examples/widgets/<widget_name>/<feature_name>/lv_example_<widget_name>_<feature_name>.xml`
- **Format**: Always wrapped in `<screen><view>...</view></screen>`
- **Layout**: Use simple, direct placement; avoid container-flow stacking for every demo item unless layout is the subject

### Doc Integration
- Add **inline feature section references** using `<LvglXMLExample name="..." path="..." />`
  - Place these immediately after the section heading or introduction text
  - Example: "### Value and range" followed by the XML example component
  - Each example must sit in the subsection that actually describes the feature it
    demonstrates. If no relevant subsection exists, create one with a short intro
    paragraph before the `<LvglXMLExample>`.
- **Do not add `<LvglExample>` list entries** for widgets whose hand-written C
  examples have been removed. For these widgets the `## Examples` section is
  deleted entirely — every example surfaces through the inline `<LvglXMLExample>`
  in its feature subsection.
- The widget's top-level `lv_example_<topic>.h` header is regenerated by
  `scripts/generate_examples.py` from whatever XML files are present in the
  topic subtree; don't hand-edit it.

## XML Markup Conventions

### Styling
- **Shared styles**: Define `<styles>` block at document root when multiple items reuse the same properties
  - Avoids repetition across similar UI elements
  - Example: grid cells with identical background, padding, radius
- **Style naming**: All style names must be prefixed with `style_` (e.g., `style_cell`, `style_label_bg`)
- **Inline style tags**: Format consistently as 3-line blocks:
  ```xml
  <lv_widget>
    <style name="style_name" />
  </lv_widget>
  ```
- **Shared colors**: Define reused colors in a `<consts>` block and reference them via
  `#name` so a single value drives every label/highlight in the example.
  ```xml
  <consts>
    <color name="text_color" value="0x013992" />
  </consts>
  <view ...>
    <lv_label style_text_color="#text_color" text="..." />
  </view>
  ```
  The `text_color` const (`0x013992`, LVGL blue) is **only** for labels embedded
  inside fill-style widgets — `lv_bar`, `lv_slider`, and similar widgets whose
  indicator paints over the label area so the default text color disappears.
  Do **not** add the const or `style_text_color` to labels in button, checkbox,
  LED, etc. examples — there the default text color renders fine.

### Layout
- **Default to column flex** for examples that stack a title plus a few demo widgets:
  ```xml
  <view flex_flow="column" style_flex_cross_place="center" style_pad_row="16">
  ```
  This centers items on the cross axis and inserts a 16 px gap between rows, so
  individual widgets do not need manual `x`/`y` coordinates.
- Use manual `x`/`y` placement only when the example specifically demonstrates
  positioning (e.g. `ignore_layout`) or when items overlap by design.

### Labels and Text
- **Place labels inside widgets** (not external siblings) when demonstrating widget features
  - Example: `<lv_arc><lv_label align="center" text="normal" /></lv_arc>`
  - Keep labels centered within the widget using `align="center"`
- **Multi-line labels**: Use `&#10;` for newlines within text attributes
- **Label text**: Keep concise; abbreviate when space-constrained (e.g., "rev." for "reverse")

### Comments and Hints
- **Top-of-file comment block**: Every example XML must begin with a multi-line comment placed *above* the `<screen>` element. It documents the example for humans, doc tooling, and AI assistants regenerating or migrating the example.
  - Format:
    ```xml
    <!--
     @title <Short capitalized example title>
     @brief <Single-sentence description ending with a period.>

     <Body: 2–3 sentences describing what the example shows, which attributes
     drive the visible difference, and why the specific values were chosen.
     This is the place to capture intent and trade-offs, not what the code
     literally does.>
    -->
    <screen>
      ...
    </screen>
    ```
  - `@title` mirrors the on-screen title label without the "XML" word (e.g. "Arc rotation offset").
  - `@brief` is a single sentence, ≤ ~80 chars, ending with a period. It is what doc generators surface first.
  - The 2–3 sentence body must:
    - Reference the *specific* attributes / values the example varies (e.g. `min_value`, `style_base_dir`, `flex_grow`).
    - Explain *why* the variation is interesting (what visual or behavioral difference it produces).
    - Be self-contained — readable without the surrounding doc page.
  - Keep wrapping consistent: hard-wrap body lines around 90 characters so the comment is readable in plain text.
- **Meaningful block comments**: Add `<!-- ... -->` above each logical demo section
  - Describe what the code demonstrates
  - Example: `<!-- Arc with negative knob offset -->`
- **Always include a 💡 hint** at the document's top-level comment, immediately after the opening `<view>`
  - Format: `<!-- 💡 Action/setting to observe and what changes. -->`
  - **Must be specific** to the feature, not generic
  - Example: ✅ "Drag each arc quickly: lower change_rate responds more slowly…"
  - Example: ❌ "Adjust the slider" (too vague)
- **Omit `<!-- Example title -->`** if a 💡 hint already exists; they are redundant
- **Comment redundancy rule**: top-of-file block describes intent, 💡 hint describes the interactive observation, inline block comments describe each demo item. Each layer should add information — do not repeat the same sentence at multiple levels.

### Flag and Property Usage
- Use lowercase for boolean/enum attribute values (e.g., `mode="normal"`, `orientation="vertical"`, `ignore_layout="true"`)
- Set only non-default properties to keep examples concise
- Verify default values in widget source before including or excluding attributes

## Layout Simplification Rules

### Container Avoidance
- Reserve manual `x`/`y` placement for examples where positioning itself is the
  subject (e.g. `ignore_layout`, overlap demos). Otherwise use the column-flex
  container described in **Layout** above so item spacing and alignment fall
  out of the layout rather than per-item offsets.

### Simple Direct Placement
- When manual placement is required, set `x` / `y` directly on the items and
  keep the structure flat — no nested wrapper containers.
- Title label sits at the top with `width="100%"` and `style_text_align="center"`.

## Example-Specific Rules

### Arc Examples
- **Avoid defaults**: Remove `bg_start_angle="135"` and `bg_end_angle="45"` unless the example explicitly deals with angles
- **Place labels inside**: Use `<lv_label align="center" text="..." />` as a child of each arc
- **Mode comparison**: Group normal, reverse, symmetrical in one file for visual comparison
- **Show variation, not repetition**: Don't show "default arc vs. arc with feature"; show two different feature values instead
  - Example: `knob_offset="-20"` vs. `knob_offset="24"` (not `0` vs. `24`)

### Slider Examples
- **Range modes**: Symmetrical requires negative+positive range; range mode uses both `left_value` and `value`
- **Orientation**: Use `orientation="vertical"` for explicit vertical; width/height can auto-detect
- **Direction reversal**: Swap min/max to reverse direction (e.g., `min_value="100" max_value="0"`)
- **Embed labels**: Place label inside slider using `align="center"` for consistency

### Flex/Grid Examples
- **Max item count**: Grid examples should have ≤ 6 items to keep visual simple
- **Grid property names**: Use `style_layout="grid"` (not `layout`)
- **Ignore layout**: Use `ignore_layout="true"` to exclude items from layout flow

### Label Examples
- **Long mode**: Consolidate all long-mode variants in one example file
- **Recolor syntax**: `#RRGGBB color# text# #RRGGBB ...` format
- **Text selection**: Use `text_selection_start` and `text_selection_end` attributes (integer indices)

### Styling Examples
- **One styling example per widget** under `<widget>/<widget>_styling/`. Goal is
  to demonstrate the widget's part/state model — *not* every style property.
- **Use named `<style>` blocks** in `<styles>`, even for one-off looks. The
  named-style + `selector` form is what readers come to learn; local `style_*`
  props on the widget tag are a separate concept and should appear only when
  illustrating MAIN-only tweaks.
- **Cover the parts the widget actually has**, with `selector="part"`:
  - Multi-part widgets (`lv_buttonmatrix`, `lv_scale-section`, `lv_slider`,
    `lv_checkbox`, …) need a style per relevant part — `main`, `items`,
    `indicator`, `knob`, `cursor`, etc.
  - **Scale section needs all three**: `style_main`, `style_items`,
    `style_indicator`. With only one of them, the band is partially invisible.
- **Show state interaction** with `selector="part|state"` (e.g.
  `items|checked`, `pressed|knob`) — that's the headline value of the
  selector grammar.
- **Pair styles with realistic values**. Use a coherent palette
  (one accent + one neutral + one text color) so the example reads as a
  finished component, not a swatch.
- **Avoid stacking variants**. One styled instance is enough; readers compare
  against the default-styled widget elsewhere on the page, not against five
  near-identical tiles in the same example.

### Data Binding Examples
- **One example per binding concept per widget**, placed in its own folder
  under the widget. Concept = `bind_value`, `bind_text`, `bind_checked`,
  `bind_style`, `bind_flag_if_*`, `bind_state_if_*`, or the event helpers.
- **Reuse the generic subjects** declared in
  `examples/xml_project/globals.xml`. Don't invent per-example subjects:
  - `subject_value` (int, 0..100, default 50) — generic numeric value
  - `subject_value2` (int, 0..100, default 20) — second numeric value
  - `subject_index` (int, default 0) — for short option lists (dropdown/roller)
  - `subject_flag` (int, default 0) — bool-like toggles
  - `subject_text` (string, default "Hello") — text payload
  - Add a new subject to `globals.xml` only if none of the above fits.
- **Make the wire visible.** Pair the bound widget with a `<lv_label
  bind_text="..." bind_text-fmt="..."/>` showing the live value. The binding
  is obvious to a reader when they can watch the number change.
- **Hold-to-repeat pattern**: for +/- increment buttons, wire *two*
  `<subject_increment_event>` children on the same button — one with
  `trigger="clicked"` and one with `trigger="long_pressed_repeat"`. Tap
  steps once, hold repeats. Single-trigger increment feels stiff.
- **Nest the controls inside the visualizer when the geometry is meaningful**.
  A spinbox + +/- buttons placed *inside* an `lv_arc` use the arc both as
  indicator and container — no flex-row wrapper, no extra panel. Reach for
  this when the widget's shape *is* the layout.
- **Escape comparison operators in attribute strings.** A raw `>` or `<`
  inside `bind_text-fmt`, `text`, or other attributes breaks XML parsing. Use
  `&gt;`/`&lt;` — or rephrase ("disabled above 80" instead of
  "disabled when > 80").
- **💡 hints describe the interaction**, not the wiring. Tell the reader what
  to do and what to watch for: "Drag the slider; the label updates because
  both widgets share `subject_value`." Never just "Adjust the slider".
- **Style/state/flag binding examples must show the reaction.** A
  `bind_state_if_gt` example needs a control that drives the subject *plus*
  the target widget that visibly disables/enables — otherwise there is
  nothing to observe.
- **Cross-widget binding belongs on the *driving* widget's page**. A slider
  that disables a button via `bind_state_if_gt` is a slider feature; the
  example lives under `slider/`, not `button/`.

## Documentation Best Practices

### Section Placement in Docs
- Inline examples (`LvglXMLExample`) go in **Usage** section under the relevant subsection
- Full example list entries (`LvglExample`) go in the **Examples** section at document end
- Both ensure users see quick reference in feature context and full reference in examples list

### Example Naming in Docs
- Titles: "Widget: feature name" format (no "XML" in title)
- Example: "### Slider: modes" or "### Arc: knob offset"

## Validation Checklist

Before finalizing examples:
- ✅ Top-of-file comment block with `@title`, `@brief`, and a 2–3 sentence body is present above `<screen>`
- ✅ Body of the top-of-file comment names the specific attributes/values that vary
- ✅ No default values are set unnecessarily
- ✅ All hints are specific to the feature, not generic
- ✅ Style names are prefixed with `style_`
- ✅ Labels are inside (not outside) widgets
- ✅ Comments exist above meaningful blocks
- ✅ No `<!-- Example title -->` when 💡 hint is present
- ✅ Default column-flex container used (or manual x/y justified by the feature)
- ✅ Shared colors defined in `<consts>` and referenced via `#name`
- ✅ Inline `<LvglXMLExample>` added in the relevant feature subsection (no
   `<LvglExample>` list entries on XML-converted widgets)
- ✅ Files compile without errors (run `scripts/generate_examples.py` on the
   new XML; the generated `.c` and the topic header are updated in place)

---

## Summary

The goal is to create **focused, pedagogical XML examples** that:
1. Demonstrate one feature per file
2. Avoid repetitive content and default values
3. Begin with a top-of-file `@title` / `@brief` / 2–3 sentence body comment that documents intent
4. Include specific, actionable 💡 hints and per-section block comments
5. Use clean, readable XML with meaningful comments at each layer (file, hint, section)
6. Integrate seamlessly into widget documentation both inline and in example lists
7. Prioritize clarity and simplicity over comprehensiveness
